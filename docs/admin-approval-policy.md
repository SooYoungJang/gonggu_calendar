# ✅ 관리자 제보 승인/반려 정책

**작성일**: 2026-06-16
**버전**: v1.0
**상태**: 초안

---

## 1. 개요

본 문서는 GongGu Calendar 서비스에서 사용자가 제보한 공동구매(공구) 정보를 관리자가 검수하여 승인 또는 반려하는 기준과 절차를 정의합니다.

---

## 2. 제보 검수 흐름

```
[사용자 제보 접수]
    │
    ▼
[자동 검증 (시스템)]
    ├── 필수값 확인 (제목, URL, 날짜)
    ├── URL 유효성 검사
    ├── 날짜 역전 검사 (종료일 > 시작일)
    └── 중복 URL 검사 (contentHash)
         └─ 중복 시 → DUPLICATE 자동 지정
    │
    ▼
[PENDING 상태 저장]
    │
    ▼
[관리자 검수]
    ├── [승인] → GroupBuy 전환
    │   ├── 확정 정보 입력
    │   └── 공구 캘린더 노출
    │
    └── [반려] → 사유와 함께 반려
        └── 제보자에게 통지 (향후)
```

## 3. 승인 기준

### 3.1 필수 승인 조건 (ALL must pass)

| # | 조건 | 설명 | 검증 방법 |
|---|------|------|-----------|
| 1 | **유효한 공구** | 실제 진행 중인 공동구매여야 함 | 관리자 판단 |
| 2 | **정확한 URL** | 실제 상품/공구 페이지 링크 | URL 접속 확인 |
| 3 | **유효한 기간** | 시작일 < 종료일, 과거/현재/미래 공구 | 자동 + 수동 |
| 4 | **명확한 정보** | 제목, 설명이 구체적이고 정확함 | 관리자 판단 |
| 5 | **중복 아님** | 동일한 공구가 이미 등록되지 않음 | contentHash + 관리자 확인 |
| 6 | **적절한 이미지** | 상품/공구 관련 이미지 또는 URL | 관리자 판단 |
| 7 | **정책 위반 아님** | 금지 품목/내용이 아님 (아래 5항 참조) | 관리자 판단 |

### 3.2 승인 가이드라인

| 요소 | 적합 예시 | 부적합 예시 |
|------|-----------|-------------|
| **제목** | "네이버 쇼핑 11번가 빅세일" | "좋은 딜" |
| **설명** | 20자 이상, 구체적인 할인 정보 포함 | "여기 좋아요" |
| **URL** | 실제 상품 페이지 (shop.naver.com, 11st.co.kr 등) | 홈페이지 메인, 검색 결과 페이지 |
| **기간** | 구체적 시작일/종료일 | "상시" 또는 "조만간" |
| **이미지** | 상품 이미지, 배너, 할인율 표기 이미지 | 개인 사진, 저작권 문제 이미지 |

## 4. 반려 기준

### 4.1 자동 반려 사유

| 사유 코드 | 설명 | 비고 |
|-----------|------|------|
| `DUPLICATE_URL` | 이미 등록된 URL (contentHash 일치) | 시스템 자동 감지 |
| `INVALID_DATE` | 날짜 역전 (종료일 < 시작일) | 시스템 자동 감지 |
| `INVALID_URL` | 잘못된 URL 형식 | 시스템 자동 감지 |
| `MISSING_FIELDS` | 필수 필드 누락 | 시스템 자동 감지 |

### 4.2 수동 반려 사유 (관리자 선택)

| 사유 코드 | 설명 | 예시 |
|-----------|------|------|
| `NOT_A_GROUP_BUY` | 공동구매가 아님 | 단순 할인 정보, 개인 거래 |
| `DUPLICATE_CONTENT` | 동일한 공구가 이미 등록됨 (URL 달라도 내용 중복) | 같은 딜 다른 링크 |
| `INACCURATE_INFO` | 정보가 부정확함 | 가격/날짜/할인율이 실제와 다름 |
| `EXPIRED` | 이미 종료된 공구 | 종료일이 과거 |
| `INAPPROPRIATE_CONTENT` | 부적절한 내용 | 광고성, 스팸, 사기 의심 |
| `INSUFFICIENT_INFO` | 정보 부족 | 설명 불충분, 이미지 없음 |
| `VIOLATES_POLICY` | 정책 위반 | 금지 품목, 성인 콘텐츠 |
| `OTHER` | 기타 사유 | 관리자 자유 입력 (필수) |

## 6. 승인 후 GroupBuy 전환

### 6.1 자동 생성 필드

| 필드 | 출처 | 설명 |
|------|------|------|
| `title` | 제보 제목 (편집 가능) | 최종 공구 제목 |
| `description` | 제보 설명 (편집 가능) | 최종 공구 설명 |
| `productUrl` | 제보 URL | 상품 페이지 |
| `imageUrl` | 제보 이미지 (편집 가능) | 대표 이미지 |
| `startDate` | 제보 시작일 (편집 가능) | 공구 시작일 |
| `endDate` | 제보 종료일 (편집 가능) | 공구 종료일 |
| `influencerId` | 관리자 선택 (필수) | 인플루언서 연결 |
| `category` | 관리자 선택 (선택) | 카테고리 |
| `status` | `ACTIVE` | 공구 상태 |

### 6.2 승인 폼 필드

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| 확정 제목 | text | ✅ | 자동 채움, 수정 가능 |
| 확정 이미지 URL | url | ❌ | 별도 이미지 지정 가능 |
| 인플루언서 | select | ✅ | 등록된 인플루언서 중 선택 |
| 카테고리 | select | ❌ | 카테고리 선택 |
| 관리자 메모 | textarea | ❌ | 내부 메모 (비공개) |

## 7. 감사 로그 (Audit Trail)

### 7.1 기록 항목

| 항목 | 설명 |
|------|------|
| 제보 ID | 대상 제보 |
| 관리자 ID | 처리한 관리자 |
| 액션 | 승인 / 반려 / 상태 변경 |
| 변경 전 상태 | PENDING → |
| 변경 후 상태 | → APPROVED / REJECTED / DUPLICATE |
| 사유 코드 | 반려 사유 (반려 시) |
| 관리자 메모 | 내부 메모 |
| 처리 일시 | ISO 8601 타임스탬프 |
| IP 주소 | 관리자 IP (보안 감사) |

### 7.2 보존 정책

| 항목 | 보존 기간 |
|------|-----------|
| 승인 이력 | 영구 |
| 반려 이력 | 1년 |
| 자동 반려 로그 | 3개월 |

## 8. 권한 및 접근 제어

### 8.1 관리자 역할

| 역할 | 권한 | 설명 |
|------|------|------|
| **뷰어** | 조회만 가능 | 대시보드/제보 목록 조회 |
| **검수자** | 승인/반려 가능 | 제보 검수 처리 |
| **슈퍼 관리자** | 모든 권한 | 설정, 계정 관리 |

### 8.2 인증 방식

- 관리자 페이지: **x-admin-token** 헤더 인증
- 토큰은 API 서버 환경 변수에 설정
- 모든 관리자 액션은 감사 로그에 기록

## 9. SLA (Service Level Agreement)

| 항목 | 목표 | 심각도 |
|------|------|--------|
| PENDING 제보 최초 검수까지 시간 | < 24시간 | 일반 |
| 긴급 제보 (대규모 할인) 검수 | < 4시간 | 긴급 |
| 승인 후 캘린더 반영 시간 | < 5분 | 일반 |
| 반려 이의제기 응답 | < 48시간 | 일반 |

## 10. 향후 개선 사항

1. **제보자 알림**: 승인/반려 시 이메일 또는 푸시 알림
2. **이의제기 시스템**: 반려된 제보에 대한 이의제기 및 재검토
3. **자동 검수 AI**: 유사 제보 탐지, 부적절 콘텐츠 자동 분류
4. **배치 승인**: 대규모 딜 시즌 대비 일괄 승인 기능
5. **제보자 평판 시스템**: 신뢰할 수 있는 제보자 가중치 부여
